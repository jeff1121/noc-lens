# 功能規格：noc-lens 排程式網路設備監控與 AI 摘要報告

**Feature Branch**: `001-network-device-monitoring`

**Created**: 2026-06-24

**Status**: Draft

**Input**: User description: "建立一個跨平台桌面版應用程式 noc-lens（Scheduled network device monitoring with AI-generated summaries and reports）。能透過 SSH 連線網路設備查詢日常維護資訊並由 AI Agent 產生摘要與報告；能匯入與管理設備清單（IPAddress、Username、Password、note）；支援自訂群組／Tag（例：高雄三民區、高雄高中）；支援品牌 Cisco、Aruba、Fortigate-ngfw、Palo Alto Networks；查詢狀態包含 CPU、Memory、module、interface、loading、traffic；能本地儲存狀態資料。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 設備清單與分組／標籤管理 (Priority: P1)

維運人員需要先把要監控的網路設備建立成一份可管理的清單。使用者可以手動新增，或從檔案匯入一批設備；每筆設備包含「IP 位址、登入帳號、登入密碼、備註」；並可為設備指定一個或多個自訂群組／標籤（例如「高雄三民區」「高雄高中」），以便依地點或用途分類檢視。

**Why this priority**: 沒有設備清單，後續所有查詢、排程與報告都無從進行。它本身也能獨立交付價值——讓使用者集中、有組織地管理散落的設備資訊。

**Independent Test**: 匯入一份含多筆設備的檔案、為其中部分設備指定群組／標籤，接著依群組篩選並檢視清單；即使不執行任何 SSH 查詢也能完成並提供價值。

**Acceptance Scenarios**:

1. **Given** 一份格式正確的設備清單檔案，**When** 使用者執行匯入，**Then** 系統建立對應的設備並顯示成功筆數與失敗（含原因）筆數。
2. **Given** 已存在的設備清單，**When** 使用者新增／編輯／刪除某筆設備，**Then** 變更立即反映在清單並保存到本地。
3. **Given** 多筆設備，**When** 使用者建立群組／標籤並指派設備，**Then** 可依該群組／標籤篩選出對應設備。
4. **Given** 設備指定了品牌，**When** 品牌不在支援清單內，**Then** 系統提示該品牌尚不支援並阻止設定為不支援的品牌。

---

### User Story 2 - 透過 SSH 即時查詢設備狀態（多品牌） (Priority: P2)

維運人員選擇一台或多台設備，立即透過 SSH 連線查詢日常維護所需的狀態資訊（CPU、Memory、module、interface、loading、traffic）。系統依設備品牌（Cisco、Aruba、Fortigate-ngfw、Palo Alto Networks）自動採用對應的查詢指令，並把回傳的原始文字重新整理成易讀的結構化結果呈現。

**Why this priority**: 這是工具的核心使用情境——讓使用者不必逐台登入即可快速取得設備健康狀態。建立在 US1 之上，但只要有一台設備即可獨立驗證。

**Independent Test**: 新增一台真實（或測試）設備，對它執行一次狀態查詢，確認 CPU／Memory／介面等資訊正確擷取並以易讀格式顯示。

**Acceptance Scenarios**:

1. **Given** 一台可連線且品牌受支援的設備，**When** 使用者發起狀態查詢，**Then** 系統回傳 CPU、Memory、module、interface、loading、traffic 的整理後結果。
2. **Given** 設備無法連線或認證失敗，**When** 使用者發起查詢，**Then** 系統明確顯示失敗原因（如逾時、認證失敗），不影響其他設備的查詢。
3. **Given** 同時選取多台設備，**When** 使用者發起查詢，**Then** 系統逐台回報各自的成功或失敗結果。
4. **Given** 某品牌不支援某項指標，**When** 查詢該指標，**Then** 系統標示「該品牌不適用」而非顯示錯誤資料。

---

### User Story 3 - 排程監控與本地狀態資料儲存 (Priority: P3)

維運人員設定排程，讓系統在指定時間或固定間隔自動對選定的設備（或群組）執行狀態查詢，並把每次結果儲存在本地，形成可回溯的歷史資料。

**Why this priority**: 將「即時查詢」升級為「持續監控」，是產品名稱「Scheduled monitoring」的核心，但需先有 US2 的查詢能力。

**Independent Test**: 對一個群組設定每日／固定間隔排程，等待或觸發一次排程執行，確認結果已自動收集並保存於本地、可於歷史中查看。

**Acceptance Scenarios**:

1. **Given** 一個群組與一個排程設定，**When** 排程時間到達，**Then** 系統自動對群組內設備執行查詢並將結果存入本地。
2. **Given** 已收集的歷史資料，**When** 使用者檢視某設備的歷史，**Then** 系統依時間顯示歷次狀態快照。
3. **Given** 排程執行期間部分設備失敗，**When** 排程完成，**Then** 系統記錄成功與失敗明細且不中斷整體排程。
4. **Given** 應用程式重新啟動，**When** 使用者開啟，**Then** 先前的設備、群組、排程與歷史資料仍存在。

---

### User Story 4 - AI 摘要與報告生成 (Priority: P3)

維運人員針對單次查詢結果或一段期間的歷史資料，請 AI Agent 產生易讀的摘要與維護報告（例如：點出異常指標、整體健康概況、建議關注項目），並可匯出報告供存檔或分享。

**Why this priority**: AI 摘要是本產品的差異化價值，但仰賴已收集的狀態資料（US2／US3）才有意義。

**Independent Test**: 在已有一批狀態資料的情況下，產生一份 AI 摘要報告並匯出，確認報告內容對應實際資料且可讀。

**Acceptance Scenarios**:

1. **Given** 一批設備狀態資料，**When** 使用者請求產生摘要，**Then** 系統輸出涵蓋整體健康狀況與異常重點的可讀摘要。
2. **Given** 一段期間的歷史資料，**When** 使用者產生報告，**Then** 報告包含趨勢觀察與需關注的設備清單。
3. **Given** 一份已產生的報告，**When** 使用者匯出，**Then** 系統輸出可保存的報告檔案。
4. **Given** AI 服務暫時不可用，**When** 使用者請求摘要，**Then** 系統明確提示無法產生並保留原始狀態資料不受影響。

---

### Edge Cases

- 匯入檔案欄位缺漏、格式錯誤或含重複 IP 時，如何回報與處理（部分成功）？
- 設備密碼為空或錯誤、SSH 連線逾時、設備拒絕連線時的行為與訊息。
- 設備回傳的指令輸出格式與預期不符（韌體版本差異）時，如何避免解析錯誤資料。
- 大量設備（如數百台）同時查詢或排程時的併發控制與整體耗時。
- 兩個排程在同一時間觸發、或前一次排程尚未結束又到下一次觸發時的處理。
- 同一設備被指派到多個群組時，排程與報告如何避免重複計算。
- 本地儲存空間不足或歷史資料持續累積時的保留策略。
- AI 服務逾時、額度用盡或回傳不完整內容時的降級處理。

## Requirements *(mandatory)*

### Functional Requirements

#### 設備清單與分組

- **FR-001**: 系統 MUST 允許使用者手動新增、編輯、刪除設備，每筆設備至少包含 IP 位址、登入帳號、登入密碼、備註（note）。
- **FR-002**: 系統 MUST 允許使用者從檔案匯入批次設備清單，並回報成功與失敗（含失敗原因）筆數。
- **FR-003**: 系統 MUST 允許使用者建立自訂群組／標籤（如「高雄三民區」「高雄高中」），且單一設備 MUST 可同時隸屬多個群組／標籤。
- **FR-004**: 使用者 MUST 能依群組／標籤篩選與檢視設備清單。
- **FR-005**: 系統 MUST 要求每筆設備標註品牌，且品牌限定於 Cisco、Aruba、Fortigate-ngfw、Palo Alto Networks。

#### SSH 查詢與狀態擷取

- **FR-006**: 系統 MUST 能透過 SSH 以設備清單中的帳號／密碼連線至設備。
- **FR-007**: 系統 MUST 依設備品牌套用對應的查詢指令集，取得 CPU、Memory、module、interface、loading、traffic 等狀態。
- **FR-008**: 系統 MUST 將設備回傳的原始文字輸出整理為結構化、易讀的結果呈現給使用者。
- **FR-009**: 系統 MUST 僅執行唯讀的查詢操作，不得對設備進行任何組態變更（v1 範圍）。
- **FR-010**: 對於某品牌不適用之指標，系統 MUST 明確標示「不適用」而非呈現錯誤或臆測資料。
- **FR-011**: 當連線或認證失敗時，系統 MUST 顯示明確的失敗原因，且單台失敗不得中斷其他設備的查詢。
- **FR-012**: 使用者 MUST 能對單台或多台（含整個群組）設備發起即時查詢。

#### 排程與本地儲存

- **FR-013**: 系統 MUST 允許使用者對設備或群組設定自動執行查詢的排程。
- **FR-014**: 系統 MUST 在排程觸發時自動執行查詢並將結果以時間戳記保存於本地。
- **FR-015**: 系統 MUST 在本地持久化保存設備、群組／標籤、排程設定與歷史狀態資料，於應用程式重啟後仍可存取。
- **FR-016**: 使用者 MUST 能檢視單一設備的歷史狀態快照（依時間排序）。
- **FR-017**: 系統 MUST 記錄每次排程執行的成功／失敗明細。

#### AI 摘要與報告

- **FR-018**: 系統 MUST 能依使用者請求，針對選定的狀態資料（單次或一段期間）產生可讀的 AI 摘要。
- **FR-019**: AI 摘要 MUST 至少涵蓋整體健康概況與異常／需關注項目。
- **FR-020**: 使用者 MUST 能將產生的報告匯出為可保存的檔案。
- **FR-021**: 當 AI 服務不可用時，系統 MUST 明確提示且不得影響既有狀態資料。
- **FR-022**: AI 摘要與報告所引用的資料 MUST 可對應回實際收集到的狀態資料來源（具可追溯性）。

#### 安全性

- **FR-023**: 系統 MUST 將設備登入密碼以加密方式保存於本地（不得以明文儲存）。
- **FR-024**: 系統 MUST 避免在介面、日誌或匯出報告中以明文顯示設備密碼。

#### 跨平台與一致性

- **FR-025**: 應用程式 MUST 以跨平台桌面應用形式提供（至少涵蓋主流桌面作業系統）。
- **FR-026**: 介面 MUST 維持一致的視覺語言，使設備狀態（正常／警告／嚴重）能被快速辨識。

#### AI 處理位置（已於規劃階段決定）

- **FR-027**: 系統 MUST 透過「可設定的 OpenAI 相容 API 端點」（base URL + API 金鑰）進行 AI 摘要與報告生成；預設可指向雲端服務，使用者亦 MUST 能改指向本地的 OpenAI 相容服務（如 Ollama／LM Studio）以使資料不離開本機。設定畫面 MUST 清楚標示「使用雲端端點時設備狀態資料將離開本機」之隱私影響。

### Key Entities *(include if feature involves data)*

- **Device（設備）**: 受監控的網路設備。屬性包含 IP 位址、登入帳號、登入密碼（加密保存）、備註、品牌；關聯至零個或多個群組／標籤。
- **Group/Tag（群組／標籤）**: 使用者自訂的分類（如地點）。一個群組可包含多台設備，一台設備可屬於多個群組。
- **StatusSnapshot（狀態快照）**: 某次查詢在特定時間取得的設備狀態（CPU、Memory、module、interface、loading、traffic 等），關聯至來源設備與時間戳記。
- **ScheduledJob（排程工作）**: 定義對哪些設備／群組、於何時或多久執行一次查詢，含執行結果紀錄。
- **BrandProfile（品牌設定檔）**: 各支援品牌對應的查詢指令集與狀態解析規則。
- **Report（報告）**: 由 AI 依一筆或多筆狀態快照產生的摘要／報告，可匯出。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 使用者可在 5 分鐘內完成首次設備清單匯入並依群組／標籤分類完成檢視。
- **SC-002**: 對單台受支援且可連線的設備，使用者可在 30 秒內取得整理後的完整狀態結果。
- **SC-003**: 系統可對至少 100 台設備設定排程並自動完成一輪查詢與本地儲存，且不需人工介入。
- **SC-004**: 90% 以上的成功查詢，其狀態資訊（CPU、Memory、interface 等）與設備實際回報內容一致。
- **SC-005**: 使用者可在 1 分鐘內，從已收集的狀態資料產生並匯出一份 AI 摘要報告。
- **SC-006**: 應用程式重新啟動後，先前的設備、群組、排程與歷史資料 100% 保留。
- **SC-007**: 任一設備查詢失敗時，使用者可在結果中清楚看到失敗原因，且不影響其他設備的查詢完成率。

## Assumptions

- 預設設備清單匯入格式為 CSV，欄位至少對應 IPAddress、Username、Password、note，並可附帶品牌與群組／標籤欄位；其他格式（如 Excel／JSON）列為未來擴充。
- SSH 連線採用設備清單中的「帳號／密碼」進行認證（依使用者提供之欄位）；SSH 金鑰認證不在 v1 範圍。
- 設備查詢為唯讀；本工具 v1 不提供對設備下達組態變更指令的能力。
- 排程預設提供「固定間隔」與「每日固定時間」兩種常見模式即可滿足 MVP，進階 Cron 表達式列為未來擴充。
- 本地狀態資料預設長期保留；當資料量過大時的自動清理／保留策略列為未來擴充（MVP 先提供手動刪除）。
- 「跨平台桌面應用」技術上可建置於 Windows、macOS、Linux；惟 v1 的「預設發行平台」為 macOS 與 Windows（Linux 可自行建置，非預設發行目標）；行動裝置不在範圍。
- AI 整合預設採「OpenAI 相容 API 端點」單一介面，雲端與本地皆可透過更換 base URL／金鑰達成（見 FR-027），以符合 MVP、避免重複實作。
- 各品牌「不適用」之指標（例如部分設備無 module 概念）允許以「不適用」呈現，視為正常情況而非錯誤。
- 同時連線之設備數量會有合理的併發上限以保護本機資源與目標設備，預設值由實作於計畫階段決定。

## Out of Scope (v1)

- 對網路設備進行任何組態變更或設定下發。
- SSH 金鑰、雙因素或集中式認證（如 TACACS+/RADIUS）整合。
- 行動裝置版本。
- 多使用者協作、帳號權限與雲端同步。
- 自動化告警通知（Email／IM）與第三方監控系統整合。
