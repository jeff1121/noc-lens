# noc-lens 專案憲章

本憲章為 noc-lens（網路維運中心 NOC 檢視 / 儀表板工具）的最高開發準則，
凌駕其他一切實作慣例。所有規格、計畫、任務與程式碼皆須符合本憲章。

## 核心原則

### I. 程式碼品質優先（Code Quality）

程式碼必須清晰、可維護且最小化。具體不可違反之規則：

- **外科手術式變更**：每一行變更都必須可直接追溯至使用者需求。不得「順手改善」
  相鄰程式碼、註解或格式；不得重構未損壞的程式碼。
- **沿用既有風格**：即使你偏好不同寫法，也必須與既有程式碼風格一致。
- **只清理自己造成的殘留**：移除「因本次變更」而失去用途的 import／變數／函式；
  不得刪除既有的死碼，發現時只回報、不刪除。
- **無投機式抽象**：不得為單次使用的程式碼建立抽象、設定項或「彈性」。
  若 200 行能寫成 50 行，必須改寫。

**理由**：NOC 工具須長期維運，每次最小化、可追溯的變更能降低回歸風險與審查成本。

### II. 測試標準（Testing Standards）

任務必須轉化為可驗證的成功條件，並以測試守護核心行為。

- **目標導向**：「修 Bug」→「先寫可重現該 Bug 的測試，再使其通過」；
  「加驗證」→「先為非法輸入寫測試，再使其通過」。
- **關鍵路徑必測**：核心商業邏輯、資料轉換與 API 契約 MUST 具備自動化測試。
- **前端互動驗證**：前端流程 MUST 以 playwright-cli 的 head 模式（有頭瀏覽器）
  實際操作驗證，確認畫面與互動符合預期。
- **變更前後綠燈**：重構前後測試皆須通過，作為「行為未改變」的證明。

**理由**：強成功條件讓開發可獨立迭代收斂，弱條件（「能動就好」）會導致反覆釐清與返工。

### III. 使用者體驗一致性（UX Consistency）

NOC 儀表板的可讀性與一致性直接影響維運判斷，UI 不得各自為政。

- **設計依據**：所有前端設計與實作 MUST 透過 `ui-ux-pro-max` skill 取得樣式、
  配色、字型搭配與版面建議，並維持全站一致。
- **一致的視覺語言**：色彩、間距、元件、狀態（正常／警告／嚴重）的呈現須統一，
  確保維運人員能快速辨識網路狀態。
- **可存取性**：互動元件須具備清楚的焦點、對比與鍵盤可操作性。
- **回饋明確**：載入、錯誤、空資料與即時更新狀態 MUST 對使用者明確可見。

**理由**：NOC 場景下，一致且清楚的 UX 能縮短故障辨識時間、降低誤判。

### IV. 效能要求（Performance Requirements）

工具須在大量網路資料下保持即時與流暢。

- **明確指標**：每個功能須在計畫中載明可量測之效能目標（如清單渲染 60 fps、
  關鍵互動 p95 < 200ms、資料更新延遲上限）。未定指標前不得宣稱「已優化」。
- **避免不必要負載**：禁止 N+1 查詢、無界限的資料抓取與阻塞主執行緒的同步運算。
- **以量測為準**：效能宣稱 MUST 有量測佐證，不得臆測。
- **即時資料效率**：即時／輪詢更新須有節流或增量機制，避免重複全量抓取。

**理由**：NOC 工具面對高頻、大量資料，效能不足會直接削弱其維運價值。

### V. MVP 與簡約優先（MVP & Simplicity）

以最小可行產品交付，先思考再動手，絕不過度設計。

- **先思考再編碼**：實作前須明確陳述假設；若有多種解讀，須並列呈現而非自行選定；
  若有更簡單的做法，須提出；若有不清楚之處，須停下並指名提問。
- **只做被要求的事**：不得加入未被要求的功能、設定或彈性；不得為不可能發生的情境
  撰寫錯誤處理（僅在系統邊界驗證）。
- **MVP 增量交付**：依使用者故事切分，使每個故事可獨立實作、獨立驗證、獨立交付。

**理由**：本專案明確採用 MVP；過度設計會拖慢交付並增加維運負擔。

## 工具與技術約束（Tooling & Technical Constraints）

- **前端設計**：MUST 使用 `ui-ux-pro-max` skill 進行前端規劃與實作。
- **資訊搜尋**：需要即時／最新外部資訊時，MUST 使用 `Felo-Search` skill 查詢。
- **前端測試**：MUST 使用 `playwright-cli` skill 以 head 模式（有頭瀏覽器）測試前端。
- **語言**：所有文件、對話與程式註解 MUST 使用繁體中文（zh-TW）。

## 開發工作流程（Development Workflow）

- **計畫先行**：多步驟任務須先提出簡短計畫，每步附帶可驗證的檢查點：
  `1. [步驟] → 驗證：[檢查]`。
- **暴露而非隱藏困惑**：遇到不確定須主動提出，列出取捨；不得靜默地擅自決定。
- **變更可追溯**：每次提交的差異應只包含與當前需求相關的變更。
- **權衡取向**：本準則偏向謹慎而非速度；對瑣碎任務可運用判斷，但不得犧牲可追溯性。

## 治理（Governance）

- 本憲章凌駕其他開發慣例；發生衝突時以本憲章為準。
- **修訂程序**：任何修訂須以文件記錄變更內容、理由與影響，並更新版本與日期；
  涉及既有行為變更者須一併提出遷移／調整方式。
- **版本政策**（語意化版本）：
  - MAJOR：移除或重新定義原則、向後不相容之治理變更。
  - MINOR：新增原則／章節或實質擴充指引。
  - PATCH：用詞釐清、錯字修正等非語意調整。
- **合規審查**：所有規格、計畫、任務與程式碼審查 MUST 驗證是否符合本憲章；
  任何額外複雜度 MUST 有明確理由，否則應簡化。
